## Overview

Update an existing multiplier. Supports partial updates (only specified fields are modified).

## Endpoint

```http
PATCH /api/v1/multipliers/:id
```

**Alternative:** `PUT /api/v1/multipliers/:id` (same behavior)

**Authorization:** ADMIN only

## Path Parameters

<ParamField path="id" type="string" required>
  UUID of the multiplier to update
</ParamField>

## Request Body

<ParamField body="loteriaId" type="string">
  UUID of the lottery
</ParamField>

<ParamField body="name" type="string">
  Display name (2-32 characters)
</ParamField>

<ParamField body="valueX" type="number">
  Multiplier value (must be positive)
</ParamField>

<ParamField body="kind" type="string">
  Multiplier type: `NUMERO` or `REVENTADO`
</ParamField>

<ParamField body="appliesToDate" type="string">
  ISO 8601 date (or null to remove filter)
</ParamField>

<ParamField body="appliesToSorteoId" type="string">
  Sorteo UUID (or null to remove filter)
</ParamField>

<ParamField body="isActive" type="boolean">
  Enable or disable the multiplier
</ParamField>

## Soft Delete

To soft delete a multiplier, send a DELETE request:

```http
DELETE /api/v1/multipliers/:id
```

With body:

```json
{
  "isActive": false
}
```

Or to re-enable:

```json
{
  "isActive": true
}
```

## Restore

Restore a soft-deleted multiplier:

```http
PATCH /api/v1/multipliers/:id/restore
```

**No request body required.**

## Response

<ResponseField name="success" type="boolean">
  Indicates if the request was successful
</ResponseField>

<ResponseField name="data" type="object">
  Updated multiplier (see Get Multiplier for full schema)
</ResponseField>

## Snapshot Immutability

<Warning>
  **Changes do not affect existing tickets**
  
  Multipliers are frozen (`finalMultiplierX`) when tickets are created. Updating a multiplier only affects:
  - New tickets created after the update
  - Future sorteo evaluations (for REVENTADO)
  
  Existing tickets retain their original multiplier values.
</Warning>

## Best Practices

<AccordionGroup>
  <Accordion title="Use soft delete instead of permanent deletion">
    Deactivate multipliers with `isActive: false` instead of permanently deleting them.
    
    This preserves audit trails and allows restoration if needed.
  </Accordion>
  
  <Accordion title="Test changes in development first">
    Before updating production multipliers:
    1. Test in development environment
    2. Review exposure analytics
    3. Communicate changes to operators
    4. Monitor initial sales after update
  </Accordion>
  
  <Accordion title="Document value changes">
    Keep a changelog of multiplier value changes for compliance and analysis:
    
    ```
    2025-03-15: Base 80x -> 85x (promotional period)
    2025-03-31: Base 85x -> 80x (end of promotion)
    ```
  </Accordion>
</AccordionGroup>
