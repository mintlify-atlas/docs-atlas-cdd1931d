## Overview

Create multipliers that determine payout amounts for winning bets. Multipliers can be configured for standard NUMERO bets or special REVENTADO payouts.

## Endpoint

```http
POST /api/v1/multipliers
```

**Authorization:** ADMIN only

## Request Body

<ParamField body="loteriaId" type="string" required>
  UUID of the lottery this multiplier belongs to
</ParamField>

<ParamField body="name" type="string" required>
  Display name for the multiplier (2-32 characters)
  
  **Examples:**
  - "Base" (for default multiplier)
  - "Base 80x"
  - "REVENTADO Especial"
  - "Promoción Fin de Semana"
</ParamField>

<ParamField body="valueX" type="number" required>
  Multiplier value (must be positive)
  
  **Common values:**
  - Standard: 70-95
  - REVENTADO: 300-1000
  
  **Formula:** `Payout = Bet Amount × valueX`
</ParamField>

<ParamField body="kind" type="string" default="NUMERO">
  Multiplier type:
  - `NUMERO`: Standard bet multiplier
  - `REVENTADO`: Extra multiplier for matching REVENTADO bets
</ParamField>

<ParamField body="appliesToDate" type="string">
  ISO 8601 date for date-specific multipliers (null = applies always)
  
  **Use case:** Special multipliers for specific dates
</ParamField>

<ParamField body="appliesToSorteoId" type="string">
  UUID of specific sorteo (null = applies to all sorteos)
  
  **Use case:** REVENTADO multipliers for specific draws
</ParamField>

<ParamField body="isActive" type="boolean" default="true">
  Whether the multiplier is active and can be used
</ParamField>

## Multiplier Types

### Base Multiplier (NUMERO)

Standard multiplier used for all NUMERO bets in the lottery.

```json
{
  "loteriaId": "loteria-uuid",
  "name": "Base",
  "valueX": 80,
  "kind": "NUMERO",
  "isActive": true
}
```

<Info>
  **Resolution priority:** Base multipliers are resolved using a 5-level hierarchy:
  1. User multiplier override
  2. Banca loteria setting
  3. Loteria multiplier with `name="Base"`
  4. Loteria rulesJson.baseMultiplierX
  5. System default (env variable)
</Info>

### REVENTADO Multiplier

Extra multiplier for winning REVENTADO bets (matching number + color).

```json
{
  "loteriaId": "loteria-uuid",
  "name": "REVENTADO Alto",
  "valueX": 500,
  "kind": "REVENTADO",
  "isActive": true
}
```

### Sorteo-Specific REVENTADO

REVENTADO multiplier for a specific draw.

```json
{
  "loteriaId": "loteria-uuid",
  "name": "REVENTADO Especial - Sorteo 12:55",
  "valueX": 600,
  "kind": "REVENTADO",
  "appliesToSorteoId": "sorteo-uuid",
  "isActive": true
}
```

### Date-Specific Multiplier

Special multiplier for a specific date.

```json
{
  "loteriaId": "loteria-uuid",
  "name": "Promoción Navidad",
  "valueX": 85,
  "kind": "NUMERO",
  "appliesToDate": "2025-12-25T00:00:00.000Z",
  "isActive": true
}
```

## Best Practices

<AccordionGroup>
  <Accordion title="Always create a Base multiplier for each lottery">
    Every lottery should have at least one multiplier with `name="Base"` and `kind="NUMERO"` to serve as the default payout multiplier.
    
    This ensures consistent behavior and avoids falling back to environment defaults.
  </Accordion>
  
  <Accordion title="Use clear naming conventions">
    Good names help identify multipliers quickly:
    - **Base multipliers:** "Base", "Base 80x", "Estándar"
    - **REVENTADO:** "REVENTADO Alto", "REVENTADO 500x", "REVENTADO Especial"
    - **Promotional:** "Promoción Fin de Semana", "Navidad 2025"
  </Accordion>
  
  <Accordion title="Set appliesToSorteoId for REVENTADO">
    For special draws, create REVENTADO multipliers with `appliesToSorteoId` to ensure they only apply to specific sorteos.
    
    This prevents accidental application to the wrong draw.
  </Accordion>
  
  <Accordion title="Monitor payout exposure">
    Higher multipliers increase payout risk. Before creating high-value multipliers:
    1. Review current exposure for the lottery
    2. Set appropriate restriction rules
    3. Monitor sales patterns
  </Accordion>
</AccordionGroup>

## Snapshot Immutability

<Warning>
  **Multipliers are frozen at ticket creation**
  
  When a ticket is created, the `finalMultiplierX` value is captured and stored in each jugada. Updating or deleting a multiplier does **not** affect existing tickets.
  
  Only new tickets created after the change will use the updated multiplier.
</Warning>
